System Requirement

The RentACompanion system is a comprehensive web-based platform that connects customers with verified companions for various social activities including dining, events, studying, and travel. The system facilitates secure bookings, payments, and reviews while ensuring user safety through robust reporting and moderation features.

Functional Requirements

FR-1: User Management

  • Users can register using email, phone number, and password
  • Secure login and logout functionality with session management
  • Users can receive real-time notifications about account activities
  • Account status management (active, suspended, banned) based on behavior
  • Role-based access control with granular permissions and dynamic access management

FR-2: Customer Features

  • Browse companion profiles with advanced filtering options
  • View detailed companion information including photos, pricing, ratings, and availability
  • Book companions by selecting specific date, time, and duration
  • Make secure payments for confirmed bookings
  • Leave ratings and reviews after booking completion
  • Save favorite companions for quick access
  • View booking history and payment records

FR-3: Companion Features

  • Create and manage detailed public profiles
  • Upload and manage multiple profile photos (up to 10 photos)
  • Set hourly rates and update pricing as needed
  • Define weekly availability schedules with specific time slots
  • Accept or reject booking requests with optional messages
  • View earnings and payment history
  • Receive and respond to customer reviews
  • Manage languages spoken and personality traits

FR-4: Booking & Payment

  • Support for multiple booking statuses: Pending, Approved, Rejected, Paid, Completed
  • Automated price calculation based on hourly rate and duration
  • Secure payment processing with multiple methods (ABA, Card, AC)
  • Payment confirmation and receipt generation
  • Booking cancellation with refund policy enforcement
  • Payment history and transaction records

FR-5: Safety & Moderation

  • Users can report inappropriate behavior or content
  • Administrators can review reports and take appropriate action
  • Notification system informs users about important account activities
  • Companion verification process before profile activation
  • Content moderation for profiles and reviews
  • User suspension and ban capabilities for policy violations

FR-6: Administrator Features

  • Dashboard overview with dynamic sidebar navigation based on user permissions
  • Role management system to create, edit, and delete custom administrative roles (e.g., Staff, Manager)
  • Granular permission assignment (e.g., manage_users, manage_bookings) per role
  • Comprehensive system user management (Add, View, Edit, Delete)
  • Platform analytics and financial reporting visualizations
  • Global platform settings configuration (Platform Name, Fee Percentage)
  • Broadcast system for platform-wide announcements

Non-Functional Requirements

3.1 Performance Requirements

  • Page Load Time: Profile browsing and booking pages should load within 2-3 seconds
  • Concurrent Users: System must support at least 1000 concurrent users
  • Image Loading: Optimized image delivery with lazy loading and caching
  • API Response Time: API response time: under 500ms

3.2 Reliability & Availability

  • Uptime: System availability of 99.5% or higher
  • Data Integrity: Referential integrity maintained through foreign key constraints
  • Backup: Daily automated database backups with 30-day retention
  • Error Handling: Graceful error handling with user-friendly messages
  • Transaction Management: ACID compliance for all financial transactions

3.3 Usability Requirements

  • Interface Design: Clean, intuitive, and mobile-responsive interface
  • Learning Curve: New users should be able to complete a booking within 5 minutes
  • Error Messages: Clear and actionable error messages for all failures
  • Help System: Contextual help and FAQs for common tasks

3.4 Scalability & Maintainability

  • Database Design: Normalized database structure supporting future growth
  • Code Quality: Well-documented, modular code following best practices
  • Horizontal Scaling: Architecture supporting load balancing and replication

Data Dictionary

1. ROLES

Purpose: Defines user roles and access levels within the system.

Field Type Key Description
role_id INT PK Role ID
role_name VARCHAR(50) UNIQUE Admin, Customer, Companion, Staff, Manager, etc.

1.1 PERMISSIONS

Purpose: Defines specific granular module access rights that can be assigned to roles.

Field Type Key Description
permission_id INT PK Permission ID
name VARCHAR(100) UNIQUE Permission code (e.g., manage_users, manage_bookings)
description VARCHAR(255) Human-readable explanation of what the permission allows

1.2 ROLE_PERMISSIONS

Purpose: Association table mapping the many-to-many relationship between roles and specific permissions.

Field Type Key Description
role_id INT PK, FK → Roles ID of the role
permission_id INT PK, FK → Permissions ID of the granted permission

2. USERS

Field Type Key Description
user_id INT PK User ID
role_id INT FK → Roles Role of user
email VARCHAR(100) UNIQUE Login email
password VARCHAR(255) Hashed password
phone VARCHAR(20) Contact number
created_at TIMESTAMP Account created

3. CUSTOMER_PROFILES

Field Type Key Description
customer_id INT PK Customer profile ID
user_id INT FK → Users Owner account
full_name VARCHAR(100) Real name
date_of_birth DATE Date of birth (18+ required)
gender ENUM Male, Female, Other, Prefer not to say
bio TEXT Short intro
location VARCHAR(100) City
profile_photo VARCHAR(255) Image URL

4. COMPANION_PROFILES

Field Type Key Description
companion_id INT PK Companion ID
user_id INT FK → Users Owner account
display_name VARCHAR(100) Public name
date_of_birth DATE Date of birth (18+ required)
age INT Age (18-99)
gender ENUM Female, Male, Non-binary, Other
bio TEXT Description
rate_per_hour DECIMAL(10,2) Price
location VARCHAR(100) Area
languages JSON Array of languages spoken (e.g., ["English", "Spanish"])
personality_traits JSON Array of personality traits (e.g., ["Outgoing", "Friendly"])
verification_status ENUM Pending, Approved, Rejected
avg_rating DECIMAL(3,2) Average stars

5. COMPANION_PHOTOS

Field Type Key Description
photo_id INT PK Photo ID
companion_id INT FK → Companion Owner
photo_url VARCHAR(255) Image path
is_primary BOOLEAN Main photo

6. AVAILABILITY

Field Type Key Description
availability_id INT PK Slot ID
companion_id INT FK → Companion Companion
day_of_week ENUM Mon–Sun
start_time TIME Available from
end_time TIME Available until

7. BOOKINGS

Field Type Key Description
booking_id INT PK Booking ID
customer_id INT FK → Customer Who booked
companion_id INT FK → Companion Who is booked
start_time DATETIME Start time
end_time DATETIME End time
status ENUM Pending, Approved, Rejected, Paid, Completed
total_price DECIMAL(10,2) Total amount
meeting_location TEXT Where to meet

8. PAYMENTS

Field Type Key Description
payment_id INT PK Payment ID
booking_id INT FK → Booking Related booking
amount DECIMAL(10,2) Paid amount
method ENUM ABA, Card, Wing
status ENUM Pending, Paid, Refunded
paid_at TIMESTAMP Payment time

9. REVIEWS

Field Type Key Description
review_id INT PK Review ID
booking_id INT FK → Booking Related booking
rating INT 1–5 stars
comment TEXT Review text
created_at TIMESTAMP When reviewed

10. FAVORITES

Field Type Key Description
favorite_id INT PK Favorite ID
customer_id INT FK → Customer Who liked
companion_id INT FK → Companion Who is liked
created_at TIMESTAMP Date added

11. NOTIFICATIONS

Field Type Key Description
notification_id INT PK Notification ID
user_id INT FK → Users Receiver
title VARCHAR(100) Short title
message TEXT Full message
is_read BOOLEAN Read or not
created_at TIMESTAMP Sent time

12. REPORTS (Safety feature)

Field Type Key Description
report_id INT PK Report ID
reporter_id INT FK → Users Who reported
target_type ENUM User / Companion / Booking
target_id INT Target ID
reason TEXT Why reported
status ENUM Pending, Resolved
created_at TIMESTAMP Report time
